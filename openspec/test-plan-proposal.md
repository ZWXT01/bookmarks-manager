## 一、概述

#### 🔐 3.3.1 auth.test.ts — 认证模块（预估 20+ 测试）

**可直接使用 createTestDb() 的纯 DB 测试：**

| 测试场景 | 用例 |
|----------|------|
| 用户初始化 | `initUserTable` 创建默认用户（env 变量 / 默认 admin） |
| 凭据验证 | 正确用户名密码 → valid=true |
| 凭据验证 | 错误密码 → valid=false + 剩余尝试次数 |
| 密码修改 | 正确旧密码 → 修改成功 → 新密码可登录 |
| 密码修改 | 错误旧密码 → 修改失败 |
| 用户信息 | 查询已有用户 → 返回信息 |
| 用户信息 | 查询不存在用户 → 返回 null |

**IP 锁定测试（需隔离全局状态）：**

| 测试场景 | 用例 |
|----------|------|
| IP 锁定 | 连续 10 次失败 → locked=true |
| IP 锁定 | 锁定后 30 分钟内 → 拒绝登录 |
| IP 解锁 | 成功登录 → 清除记录 |
| IP 隔离 | 不同 IP 的失败记录互不影响 |

**API Token 测试：**

| 测试场景 | 用例 |
|----------|------|
| 创建 Token | 返回 token 字符串、ID、prefix |
| 创建 Token | 设置有效期 → expires_at 正确 |
| 列出 Token | 返回所有 token（不含实际值） |
| 验证 Token | 有效 token → valid=true |
| 验证 Token | 过期 token → expired=true |
| 验证 Token | 无效 token → valid=false |
| 删除 Token | 删除后验证 → valid=false |
| 清理过期 | cleanupExpiredTokens 删除过期记录 |

**约束：**
- `ipAttempts` 是模块级全局 Map，测试间需使用不同 IP 或手动重置
- `setInterval` 自动清理可用 `vi.useFakeTimers()` 控制

#### 🔍 3.3.2 checker.test.ts — 书签检查器（预估 15+ 测试）

**需 mock fetch 的测试：**

| 测试场景 | 用例 |
|----------|------|
| 正常 URL | HEAD 200 → ok=true |
| 空 URL | → ok=false, error='URL为空' |
| HEAD 405/403 | 自动 fallback 到 GET → 验证 ok |
| GET 也失败 | HEAD 405 + GET 500 → ok=false, httpCode=500 |
| 超时 | AbortError → error='超时' |
| 网络错误 | fetch 抛异常 → error='网络错误' |
| 无协议 URL | 自动添加 http:// 尝试 |
| 重试机制 | 首次失败 + 重试成功 → ok=true |
| 重试耗尽 | N 次全部失败 → 返回最后一次结果 |

**runCheckJob 集成测试（需 mock fetch + 真实 DB）：**

| 测试场景 | 用例 |
|----------|------|
| 正常执行 | N 个书签全部检查 → job 状态=done |
| skip_check | 标记跳过的书签 → skipped 计数 |
| 取消任务 | 中途设置 canceled → 停止检查 |
| 并发控制 | concurrency=3 → 最多 3 个并发 worker |
| 进度更新 | 每 10 个更新一次 job 进度 |
| 失败记录 | 检查失败 → addJobFailure 记录原因 |

**约束：**
- 必须 mock `global.fetch`，避免真实网络请求
- 重试延迟用 `vi.useFakeTimers()` 或设 retryDelayMs=0
- `runCheckJob` 涉及 DB 状态更新，需真实 DB

**各路由模块测试概要：**

| 路由文件 | 关键测试场景 |
|----------|------------|
| `auth.ts` | 登录成功/失败、登出、Token 换 Session、密码修改、Token CRUD |

- 异步任务：check/import/AI 路由启动后台任务，测试需等待或验证 job 状态
- SSE：`/jobs/:id/events` 需测试流式响应（可验证 headers + 首条消息）

## 五、约束集合汇总

### 硬约束

| # | 约束 | 影响范围 |
|---|------|----------|
| H1 | `checker.ts` 必须 mock `global.fetch` | checker 单元测试 |
| H2 | AI 模块必须 mock OpenAI SDK | ai-organize 测试 |
| H3 | `ipAttempts` 全局状态需测试间隔离 | auth 测试 |
| H4 | 快照/备份测试需临时目录 + afterEach 清理 | 路由集成测试 |
| H5 | SSE 端点测试需处理流生命周期 | jobs/check 路由 |
| H6 | 路由测试需注入 session mock 模拟已认证状态 | 所有路由测试 |
| H7 | AI Plan 生命周期有严格状态机，需覆盖所有合法/非法转换 | ai-organize-plan 测试 |
| H8 | 回滚有 24 小时 TTL 窗口，需 fake timers | ai-organize-plan 测试 |

### 软约束

| # | 约束 | 建议 |
|---|------|------|
| S1 | 重试延迟用 0ms 或 fake timers | checker/AI 测试 |
| S2 | `process.env` 依赖在测试前设置/恢复 | auth/settings 测试 |
| S3 | 并发 worker 在测试中降为 1 | checker runCheckJob |
| S4 | 路由测试用工厂模式避免插件重复注册 | 所有路由测试 |

### 风险

| 风险 | 缓解措施 |
|------|----------|
| 真实网络调用导致 Flaky | 严格 mock fetch |
| 全局状态交叉污染 | afterEach 重置 + 唯一 IP |
| Timer 导致测试慢 | fake timers + 0ms delay |
| 文件系统泄漏 | 临时目录 + finally 清理 |
| SSE 测试挂起 | 设置超时 + 主动关闭流 |
| 迁移假覆盖 | 至少验证迁移路径不报错 |
