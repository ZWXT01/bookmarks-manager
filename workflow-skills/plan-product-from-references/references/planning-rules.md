# 规划规则

本文件用于决定规划周期、阶段数、issue 粒度和 Agent 运行方式。

## 1. 何时只需要短期方案

满足以下多数条件时，可只输出短期方案：

- 只是一个局部功能
- 没有新增来源模型
- 没有新增外部系统
- 没有账号风控
- 没有明显运维复杂度

建议输出：

- `短期 roadmap`
- `小批 issue`
- 不必拆 `R1 ~ R4`

## 2. 何时需要中长期路线图

满足以下任一条件时，建议输出中长期路线图：

- 是一个完整产品，而不是单点功能
- 需要重做信息架构
- 需要拆 Reader / Admin / Worker / Collector 等多个层次
- 涉及 Telegram、Slack、微信等外部平台
- 涉及风控、账号授权、session、安全
- 涉及热点平台、内容源、多阶段接入

建议输出：

- 至少 `R1 / R1.5 / R2`
- 若存在明显验证风险，加入 `Spike`
- 若存在设备侧执行，加入 `R3` 或单独 collector 阶段

## 3. 何时必须加入 Spike / Gate

满足以下任一条件时，必须加入 `Spike` 或 `Gate`：

- 需要验证平台能力边界
- 需要评估账号风险
- 需要判断 API 是否可行
- 需要判断抓取稳定性
- 需要判断是否引入外部设备

典型例子：

- Telegram 内容源
- 本地设备采集
- Cookie / 反爬依赖的热点来源

## 4. issue 粒度规则

每条 issue 应满足：

- 单一目标
- 有明确边界
- 有可验证完成标准

不要把以下内容揉进同一个 issue：

- 同时做数据库、后端、前端、文档且没有边界
- 同时做“设计 + 原型 + 正式实现 + 联调 + 验收”
- 把一整阶段写成 1 条 issue

推荐拆法：

- `DB`
- `BE`
- `WK`
- `FE`
- `DOC`
- `QA`

## 5. 默认执行级别规则

### `A0`

- 仓库内可直接做
- 不依赖 secrets
- 不依赖外部设备

### `A1`

- 可做，但验证依赖本地环境、网络、样例数据或外部服务可达性

### `H1`

- 需要 secrets、凭证、账号授权、验证码

### `H2`

- 需要设备、操作系统实例、远程节点、长期在线 host

### `G1`

- 属于 gate 结论，不得被自动跳过

## 6. 默认状态队列规则

若要让 Code Agent 自动执行而不是人工逐条点名：

- 必须生成顺序化状态队列
- 队列顺序应与依赖顺序一致
- 不允许 Agent 跳过第一个未完成但已可执行的任务
- 不允许 Agent 越过 `blocked` 和 `G1` 任务
- 状态队列应显式包含 `started_at`、`completed_at`、`blocked_reason`
- 若任务完成，还应要求 Agent 额外留下 issue 级 Git 提交留痕

## 7. 默认留痕与风险规则

若规划结果要交给 Code Agent 自动执行，默认还应编码以下规则：

- 必须维护独立风险台账，不能只在 runbook 中口头描述风险
- 风险台账不仅是记录资产，也可以作为“风险排查执行模式”的唯一状态源
- 每完成一个 issue，默认至少产生 1 次独立 Git commit
- 正常执行时，每笔 commit 只记录 1 个 issue 编号
- 风险排查执行模式下，每笔 commit 默认只记录 1 个 `risk_id`
- 只有“补录历史上已完成但未提交的连续任务”时，才允许一次 commit 记录多个 issue 编号
- 若当前目录尚未初始化为 Git 仓库，Agent 开始自动执行前必须先创建仓库
- 若某 issue 完成后仍存在残余风险、验证盲区、兼容性缺口或待人工确认项，必须回写到独立风险台账
- 风险台账应同时区分 `risk_status` 与 `fix_status`，并记录 `fix_issue_id`、`git_commit`、`started_at`、`completed_at`、`blocked_reason`

## 8. 默认安全与高风险操作规则

若规划结果会落到真实仓库、真实环境或自动执行流程，默认必须编码以下规则：

- `.env`、真实连接串、真实 token、cookie、私钥、证书、数据库转储等敏感信息不得提交到 Git
- `.env.example`、runbook、issue、风险台账和截图中只能出现占位值、键名或脱敏信息，不能回写真实 secrets
- 涉及 secrets 的任务应明确区分“人工提供”“人工轮换”“可本地自动生成”三类，不能模糊处理
- 涉及删库、删卷、清理数据、开放公网端口、批量重启、凭证轮换等高风险操作时，issue 与 runbook 必须写清影响面、确认点、回滚方案和验收标准
- 若高风险操作会影响现网、用户数据或安全边界，默认应要求人工确认，不能把它描述成可无脑自动执行的 A0 任务
- secrets 泄露、端口暴露、误删数据、权限扩大、异常进程等运维安全问题应进入独立风险台账，而不是只写在临时说明里
- 若任务包含测试、调试或一次性验证，issue 与 runbook 必须同时写清“如何清理临时进程、临时端口、临时二进制、临时 `.env` 与测试数据”

## 9. 推荐输出层级

### 小项目

- `01-reference-analysis.md`
- `02-roadmap.md`
- `03-issue-breakdown.md`

### 中型及以上项目

完整输出：

- `01-reference-analysis.md`
- `02-roadmap.md`
- `03-issue-breakdown.md`
- `04-agent-runbook.md`
- `05-agent-status.md`
- `06-risk-log.md`

默认情况下，本 skill 应输出完整六件套。
